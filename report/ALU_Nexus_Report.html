ALU Nexus — Technical Report

A Flutter + Firebase platform connecting ALU students with student-led startups

David Muotoh-Francis · African Leadership University · Mobile Application Development — Final Project

1. Problem statement

ALU students need practical experience before graduating; ALU's student-led startups need affordable, motivated talent. Today that matching happens informally through WhatsApp groups and word of mouth — postings get lost, there is no application record, and students cannot verify that a "startup" is legitimate. ALU Nexus formalizes this marketplace: verified startups publish structured internship opportunities, students discover them through search and skill matching, and both sides track applications through a transparent pipeline.

3. Architecture decisions

3.1 Feature-first Clean Architecture

Each feature (auth, opportunities, applications, startups, notifications, admin) owns three layers: domain — plain entities with no Firebase imports, keeping business objects testable and framework-independent [1]; data — models extending the entities with Firestore (de)serialization, plus repositories owning every Firestore/Auth call; presentation — Cubits and screens. Screens never touch Firebase directly.

This isolation enables the app's demo mode: demo repositories subclass the production repositories and override their I/O methods with in-memory data, so the entire UI, routing, and state layer runs unchanged with or without a backend — concrete evidence that the layers are genuinely decoupled.

3.2 State management: Cubit (bloc 9.x)

Cubit was chosen over full BLoC events, Provider, and Riverpod. Versus full BLoC: the app's interactions are direct method calls (search(), submitApplication()); event classes would add ceremony without needed auditability [2]. Versus Provider: Cubit gives an explicit sealed state machine (Initial → Loading → Loaded | Error) rather than ad-hoc notifyListeners, making loading/error handling exhaustive at the UI layer. Versus Riverpod: a one-person team favors bloc's smaller conceptual surface and first-class Flutter integration.

One deliberate exception: bookmarks use a lightweight ChangeNotifier singleton persisted to SharedPreferences — device-local personalization does not warrant a Firestore round-trip.

3.3 Real-time propagation

Every list screen subscribes to Firestore snapshots() streams inside its Cubit. A state-changing write (e.g., a startup shortlists an applicant) triggers: Firestore commit → snapshot event → Cubit emit → BlocBuilder rebuild. No polling, no manual refresh; the Firebase Console reflects the same event simultaneously.

3.4 Navigation

GoRouter with a refreshListenable bound to AuthCubit's stream. Centralized redirect enforces: unauthenticated → login; authenticated-but-not-onboarded → role-specific onboarding; onboarded → role-specific home. Deep links cannot bypass the guard because it is declarative, not per-screen.

4. Firebase data model

users/{uid}            role, skills[], isOnboardingComplete, ...
startups/{id}          ownerId, verificationStatus, stage, focusAreas[], ...
opportunities/{id}     startupId, startupVerified, skills[], deadline, isActive, ...
applications/{id}      opportunityId, applicantId, status, statusNote, interviewDate, ...
notifications/{id}     userId, type, referenceId, isRead, ...
admin_actions/{id}     adminId, targetId, action, note, timestamp

Denormalization is intentional: opportunities embeds startupName/startupVerified, and applications embeds display fields from both sides. This trades write-time duplication for read-time efficiency — list screens render from a single query with no N+1 joins, which Firestore pricing and latency strongly favor [3]. Security rules (version-controlled in firestore.rules) enforce owner-only writes, admin gates, and applicant/startup-scoped application access; 13 composite indexes are declared in firestore.indexes.json and deployed with the app.

5. Feature workflows

Startup verification (ALU-specific): startup completes onboarding → verificationStatus: pending → admin approves/rejects with a note → decision written to the startup document and audit-logged to admin_actions → the feed query hard-filters startupVerified == true, so unverified content is excluded at the query level, not hidden client-side.

Application pipeline: duplicate submissions blocked by a pre-submission query; each status change (Reviewing → Shortlisted → Interviewing → Accepted/Rejected) optionally carries a note and interview date; students see a five-step progress visualization and can withdraw while active.

Skill matching: students select skills during onboarding (stored on users/{uid}.skills). The feed scores every active opportunity by case-insensitive overlap with those skills and features the best match ("Matches N of your skills").

6. UI/UX reasoning

  • One accent color (royal blue #2563EB) on flat white surfaces; color is reserved for meaning (green = paid/approved, amber = pending, red = deadline). Gradients and decorative effects were removed in review — they added noise without information.
  • Progressive disclosure on the feed: greeting → search + filter → category chips → one skill-matched featured card → compact tiles. The student's first question ("is there anything for me?") is answered before scrolling.
  • Pipeline visualization: application status is a five-step progress bar, not a text label — the core anxiety this app addresses is not knowing where an application stands.
  • No dead ends: every list has loading (shimmer), error (retry), and empty (guidance) states; dead menu items found in review were implemented (bookmarks) or removed.
  • Trust signals: the verified badge appears wherever a startup is named; unverified startups cannot reach the feed at all.

7. Error, loading, and edge-case handling

  • Every Cubit models Loading/Error states; screens render shimmer placeholders or retryable error panels.
  • Firebase Auth error codes mapped to human-readable messages (email-already-in-use, wrong-password, too-many-requests, …).
  • Form validation: ALU email domain, password strength, 100-character minimum cover letter, optional-but-validated URLs.
  • Duplicate applications rejected with explanation; expired opportunities disable Apply; near deadlines flagged "Closing soon"; explicit empty states throughout.

8. Scalability considerations

  • Query cost — list queries are indexed, filtered server-side, and bounded. Client-side text search over the streamed page is appropriate now; the drop-in replacement at scale is a search service (Algolia/Typesense) fed by Firestore triggers [3].
  • Skill matching — client-side scoring is O(n·m) per feed load. The repository already exposes getRecommendedOpportunities() using arrayContainsAny, so ranking can move server-side (Cloud Function + composite index) without touching the UI layer.
  • Notification fan-out — a Cloud Functions trigger on application status change centralizes fan-out while client code stays unchanged.
  • Team scale — feature-first foldering makes new features additive: new folder, new Cubit, new routes; no cross-feature edits.

9. Testing strategy

  1. Static analysis as a gate — flutter analyze at 0 issues; all deprecations resolved to current Material 3 APIs rather than suppressed.
  2. Unit tests for pure logic — 16 tests cover the validator layer (ALU-domain rules, password policy, URL/length), the highest-risk pure logic in the app.
  3. Demo mode as integration harness — demo repositories subclass production ones, so demo runs exercise every Cubit transition, route guard, and screen against deterministic data. Full student/startup/admin journeys were walked on Chrome and a physical Android device (Samsung S22 Ultra).
  4. Rule-level security — Firestore rules are version-controlled and deployed with the app, making access control reviewable rather than console-configured.

10. Challenges encountered & resolutions

ChallengeResolution
Firebase-coupled construction crashes. Early repositories captured FirebaseAuth.instance in constructors, crashing at startup without Firebase configured.Refactored to lazy getters (FirebaseAuth get _auth => FirebaseAuth.instance) — Firebase touched only on method invocation, which also enabled demo subclassing.
Demo data vs. type safety. Screens resolve Cubits by concrete type, so standalone demo cubits broke every screen.Demo cubits subclass the real ones, injecting overridden repositories through the existing constructor — zero screen changes.
Organization billing policy blocked Firestore. The ALU-managed Google Cloud org requires billing before Firestore database creation.Diagnosed via CLI (identical 403 across regions/editions ⇒ project-level policy); resolved by provisioning the Firebase project outside the org with owner access granted to the school account. Demo mode kept development unblocked meanwhile.
Compound query index requirements. Firestore rejects multi-field filtered queries until composite indexes exist — a mid-demo failure mode.All 13 composite indexes declared in firestore.indexes.json, deployed in one command instead of clicked reactively from error links.
Dart interpolation in currency text. '$200/month' parses as interpolation — a subtle syntax error.Raw strings (r'$200/month') for all currency literals.
Invisible ink-splash assertions. ListTile inside decorated Containers throws debug assertions (ripples paint on the nearest Material).Replaced decorated containers with Material(shape: …) wrappers, unifying how every card is built.

11. Lessons learned

  • Depend on interfaces, defer side effects. The lazy-getter refactor was two lines per repository but unlocked the entire demo-mode architecture.
  • Infrastructure belongs in version control. Rules and indexes as files turned console clicking into reviewable, redeployable artifacts.
  • Cloud constraints are design inputs. An org billing policy — not code — was the biggest blocker; building the app to run without its backend made it a footnote instead of a stop-the-world problem.
  • Restraint is a design skill. The strongest UI iteration removed things: gradients, emojis, dead menu items.
  • State machines surface edge cases. Modeling every list as Initial | Loading | Loaded | Error forced real loading/error/empty UI everywhere.

12. Limitations & future work

  • Push notifications (FCM) are scaffolded but not delivered end-to-end.
  • Resume/file upload to Firebase Storage is stubbed at the UI level.
  • In-app messaging between startups and shortlisted candidates is the next highest-value feature.
